# Markdown Image Kit 用户手册

## 📖 插件简介

**Markdown Image Kit** 是一款专为 JetBrains IDE 设计的 Markdown 图片管理插件。它能够帮助你在 IDE 中高效地处理 Markdown
文档中的图片，提供图片上传、图床迁移、标签转换、图片压缩等强大功能。

### 核心功能

- **🚀 图片上传**：支持粘贴上传、单个上传、批量上传
- **🔄 图床迁移**：轻松将图片从一个图床迁移到另一个，或下载到本地
- **🎨 图片增强**：支持压缩、重命名、水印、格式转换（WebP）
- **🔗 标签转换**：自动转换图片标签格式，支持自定义模板
- **📦 多图床支持**：支持 SM.MS、阿里云 OSS、七牛云、腾讯云 COS、GitHub、Gitee、PicList 等

---

## ⚙️ 设置页面详解

打开设置：`Settings/Preferences` → `Markdown Image Kit`

设置页面分为三个主要部分：

1. **图片处理设置**
2. **图片增强处理设置**
3. **上传服务设定**

## 📁 图片处理设置

### 1. 插入图片时的操作

#### 配置项说明

此设置控制在 Markdown 文档中插入图片时的默认行为，提供以下选项：

| 选项                                 | 说明                                 | 生成的图片路径格式                           |
|------------------------------------|------------------------------------|-------------------------------------|
| **无特殊操作**                          | 不做任何处理                             | 保持原始路径                              |
| **复制图片到当前文件夹**                     | 将图片复制到 Markdown 文件所在目录             | `./image.png`                       |
| **复制图片到 ./assets 文件夹**             | 将图片复制到当前目录下的 assets 子文件夹           | `./assets/image.png`                |
| **复制图片到 ./${filename}.assets 文件夹** | 将图片复制到以 Markdown 文件名命名的 assets 文件夹 | `./${filename}.assets/image.png`    |
| **上传图片**                           | 自动上传到配置的图床                         | `https://cdn.example.com/image.png` |
| **复制到指定路径**                        | 将图片复制到自定义路径                        | 根据输入的路径决定                           |

#### 适用场景

##### 场景一：本地文档编辑（推荐使用"复制图片到 ./assets 文件夹"）

- **适合**：文档存储在 Git 仓库，需要将图片一并提交
- **优点**：图片和文档在一起，易于管理和迁移
- **示例**：技术笔记、项目文档、开源项目 README

##### 场景二：博客写作（推荐使用"上传图片"）

- **适合**：最终需要发布到网络平台
- **优点**：图片托管到 CDN，访问速度快，不占用本地空间
- **示例**：个人博客、技术文章、公众号内容

##### 场景三：Typora 兼容（推荐使用"复制图片到 ./${filename}.assets 文件夹"）

- **适合**：与 Typora 编辑器的默认行为保持一致
- **优点**：多个编辑器之间切换使用
- **示例**：团队协作，部分成员使用 Typora

##### 场景四：灵活路径管理（使用"复制到指定路径"）

- **适合**：有特殊的目录结构要求
- **优点**：完全自定义图片存储位置
- **示例**：多个 Markdown 文件共享图片库

#### 自定义路径说明

当选择"复制到指定路径"时，会显示自定义路径输入框。路径支持以下格式：

- **相对路径**：`./images`、`../shared/imgs`
- **占位符**：`./imgs/${filename}` （文件名为 Markdown 文件名）

**示例**：

- 输入 `./images`：图片保存到 `./images/image.png`
- 输入 `../assets`：图片保存到上级目录的 assets 文件夹

### 2. 应用规则设置

#### 对本地位置的图片应用上述规则

- **作用**：将本地图片按照"插入图片时的操作"规则进行处理
- **适用场景**：
    - 整理现有文档，将散落的本地图片统一归档
    - 重构项目，调整图片存储位置
    - 批量处理从其他编辑器迁移过来的文档

**示例**：
假设文档中有图片 `![](C:/Users/Desktop/screenshot.png)`，开启此选项后：

- 如果设置为"复制到 ./assets"：图片会被复制到 `./assets/` 目录，标签更新为 `![](./assets/screenshot.png)`

#### 对网络位置的图片应用上述规则

- **作用**：将网络图片按照"插入图片时的操作"规则进行处理
- **适用场景**：
    - 图床即将到期，需要下载保存到本地
    - 引用的外部图片不稳定，需要本地化
    - 统一图片管理，避免依赖多个外部链接

**示例**：
假设文档中有图片 `![](https://old-cdn.com/image.png)`，开启此选项后：

- 如果设置为"复制到 ./imgs"：图片会被下载到 `./imgs/` 目录，标签更新为 `![](./imgs/image.png)`
- 如果设置为"上传图片"：图片会被下载后重新上传到你配置的图床

> ⚠️ **注意**：开启此选项会下载网络图片，请确保网络连接稳定，且有足够的本地存储空间。

### 3. 图片语法偏好

#### 优先使用相对路径

- **作用**：关闭后会使用绝对路径插入图片
- **适用场景**：
    - ✅ **开启**（推荐）：文档需要在不同环境中使用，具有更好的可移植性
    - ❌ **关闭**：文档仅在固定环境使用，需要精确的文件系统路径

**示例**：

- 开启：`![](./imgs/image.png)` （可移植）
- 关闭：`![](/Users/username/project/imgs/image.png)` （固定路径）

#### 为相对路径添加 ./

- **作用**：在相对路径前添加 `./` 前缀
- **适用场景**：
    - 符合某些 Markdown 渲染器的要求
    - 使路径语义更清晰，明确表示"当前目录"

**示例**：

- 开启：`![](./imgs/image.png)`
- 关闭：`![](imgs/image.png)`

> 💡 **提示**：只有在开启"优先使用相对路径"后，此选项才会生效。

#### 插入时自动转义图片 URL

- **作用**：自动对图片 URL 中的特殊字符进行转义
- **适用场景**：
    - 图片文件名包含空格：`image file.png` → `image%20file.png`
    - 图片文件名包含中文：`示例图片.png` → `%E7%A4%BA%E4%BE%8B%E5%9B%BE%E7%89%87.png`
    - 图片 URL 包含特殊字符：`#`、`&`、`?` 等

**示例**：

- 开启：`![](./imgs/image%20with%20space.png)` ✅
- 关闭：`![](./imgs/image with space.png)` ❌ 可能无法正确显示

> 💡 **建议**：推荐开启此选项，避免因特殊字符导致图片无法显示。

## 🎨 图片增强处理设置

### 1. 图片重命名

#### 配置项说明

- **开关**：勾选后启用图片重命名功能
- **重命名模板**：支持占位符自定义文件名格式

#### 支持的占位符

| 占位符                  | 说明          | 示例                                  |
|----------------------|-------------|-------------------------------------|
| `${filename}`        | 原文件名（不含扩展名） | `screenshot`                        |
| `${datetime:format}` | 日期时间格式化     | `${datetime:yyyyMMdd}` → `20251102` |
| `${string:length}`   | 随机字符串       | `${string:6}` → `aB3xK9`            |
| `${number:length}`   | 随机数字        | `${number:6}` → `123456`            |

#### 常用模板示例

| 模板                                 | 生成示例                      | 适用场景    |
|------------------------------------|---------------------------|---------|
| `${filename}`                      | `screenshot.png`          | 保持原文件名  |
| `${datetime:yyyyMMdd}-${filename}` | `20251102-screenshot.png` | 按日期归档   |
| `${datetime:yyyyMMdd_HHmmss}`      | `20251102_153045.png`     | 精确时间戳   |
| `MIK-${string:6}`                  | `MIK-a3Xk9p.png`          | 避免文件名冲突 |
| `img-${number:6}`                  | `img-123456.png`          | 数字编号    |
| `${datetime:yyyy/MM}/${filename}`  | `2025/11/screenshot.png`  | 按年月分目录  |

#### 适用场景

##### 场景一：保持原文件名（`${filename}`）

- **适合**：文件名已经有意义，不需要修改
- **示例**：设计稿、产品截图（文件名是功能描述）

##### 场景二：按日期归档（`${datetime:yyyyMMdd}-${filename}`）

- **适合**：需要按时间顺序管理图片
- **示例**：日志记录、每日报告、项目进度文档

##### 场景三：避免冲突（`MIK-${string:6}`）

- **适合**：多人协作，避免文件名重复
- **示例**：团队共享图床、公共文档库

##### 场景四：批量处理（`img-${number:6}`）

- **适合**：批量插入图片，需要统一命名
- **示例**：教程文档、产品介绍

### 2. 图片压缩

#### 配置项说明

- **开关**：勾选后启用图片压缩功能
- **压缩质量**：40-90，数值越小压缩率越高，文件越小，但质量会下降

#### 压缩效果参考

| 压缩质量 | 压缩率       | 效果   | 适用场景           |
|------|-----------|------|----------------|
| 90   | 低（~20%）   | 几乎无损 | 重视质量，对文件大小不敏感  |
| 80   | 中（~40%）   | 轻微损失 | **推荐**，质量与大小平衡 |
| 60   | 高（~60%）   | 明显损失 | 对文件大小敏感，质量要求不高 |
| 40   | 很高（~70%+） | 质量较差 | 仅用于临时预览        |

#### 适用场景

##### 场景一：博客写作（推荐 70-80）

- **需求**：减少加载时间，提升用户体验
- **说明**：网络图片需要考虑加载速度

##### 场景二：移动端访问优化（推荐 60-70）

- **需求**：节省流量，提升移动端加载速度
- **说明**：移动设备屏幕较小，对图片质量要求相对较低

##### 场景三：存储成本控制（推荐 60）

- **需求**：减少存储空间占用
- **说明**：图床存储有限制或按流量收费

##### 场景四：高清图片存档（不推荐压缩或 90）

- **需求**：保留图片细节
- **说明**：设计稿、高清截图、重要资料

> ⚠️ **注意**：
> - 支持 PNG、JPG、JPEG、BMP 格式
> - GIF 格式暂不支持（会保留原始动图）
> - 压缩不可逆，建议保留原图备份

### 3. 转为 WebP

#### 配置项说明

- **开关**：勾选后自动将图片转换为 WebP 格式
- **WebP 质量**：40-90，控制 WebP 压缩质量

#### WebP 格式优势

- 🚀 文件更小（相同质量下比 PNG 小 26%，比 JPEG 小 25-34%）
- ✅ 支持透明度（替代 PNG）
- ✅ 支持动画（替代 GIF）
- ✅ 现代浏览器广泛支持

#### 适用场景

##### 场景一：网页发布（推荐开启）

- **需求**：减少页面加载时间，提升 SEO 排名
- **说明**：现代浏览器都支持 WebP

##### 场景二：移动端应用（推荐开启）

- **需求**：节省流量和存储空间
- **说明**：移动端对文件大小更敏感

##### 场景三：存储成本优化（推荐开启）

- **需求**：减少图床存储和流量费用
- **说明**：大量图片场景下效果明显

##### 场景四：兼容性要求高（不推荐开启）

- **需求**：需要在旧浏览器或特定环境中使用
- **说明**：某些旧设备或特定软件不支持 WebP

> 💡 **建议**：如果你的文档主要在现代浏览器中查看，推荐开启 WebP 转换，质量设置为 75-85。

### 4. 添加水印

#### 配置项说明

- **开关**：勾选后为上传的图片添加文字水印
- **水印文本**：自定义水印内容，如 `@YourName`、`© 2025`

#### 适用场景

##### 场景一：版权保护

- **需求**：防止图片被盗用
- **示例**：原创设计稿、摄影作品

##### 场景二：品牌宣传

- **需求**：增加品牌曝光度
- **示例**：公司博客、产品截图

##### 场景三：标识来源

- **需求**：标明图片出处
- **示例**：教程文档、技术分享

##### 场景四：纯文档图片（不推荐开启）

- **说明**：代码截图、配置界面等不需要水印

> ⚠️ **注意**：当前版本水印功能较基础，复杂需求建议使用 PicList 的水印插件。

### 5. 替换为 HTML 标签

#### 配置项说明

- **开关**：勾选后将 Markdown 原始标签转换为 HTML 标签
- **标签类型**：原始、正常、点击放大、自定义

#### 标签类型说明

##### 原始（默认）

```markdown
![](image.png)
```

- **适用场景**：标准 Markdown 文档，不需要特殊效果

##### 正常

```html
<a href="image.png"><img src="image.png"/></a>
```

- **效果**：点击图片在新标签页打开
- **适用场景**：
    - 博客文章，方便读者查看原图
    - 产品文档，需要查看高清图片

##### 点击放大

```html
<img src="image.png" data-fancybox="gallery"/>
```

- **效果**：点击图片在当前页面放大显示（需要配合 fancybox 库）
- **适用场景**：
    - VuePress、Hexo 等静态博客
    - 在线文档平台

**VuePress 配置示例**：
在 `.vuepress/config.js` 的 `head` 中添加：

```javascript
module.exports = {
  head: [
    ['script', { src: 'https://cdnjs.cloudflare.com/ajax/libs/jquery/3.3.1/jquery.slim.min.js' }],
    ['script', { src: 'https://cdnjs.cloudflare.com/ajax/libs/fancybox/3.5.2/jquery.fancybox.min.js' }],
    ['link', { rel: 'stylesheet', href: 'https://cdnjs.cloudflare.com/ajax/libs/fancybox/3.5.2/jquery.fancybox.min.css' }]
  ]
}
```

##### 自定义

- **说明**：使用自定义模板生成标签
- **占位符**：
    - `{{0}}` 或 `${title}` - 图片标题（alt 文本）
    - `{{1}}` 或 `${path}` - 图片地址（URL）

**自定义模板示例**：

1. **带样式的图片**

```html
<img alt="{{0}}" src="{{1}}" style="max-width:100%;border:1px solid #ddd;"/>
```

2. **图片说明文字**

```html
<figure>
  <img src="{{1}}" alt="{{0}}"/>
  <figcaption>{{0}}</figcaption>
</figure>
```

3. **响应式图片**

```html
<picture>
  <source srcset="{{1}}" type="image/webp">
  <img src="{{1}}" alt="{{0}}">
</picture>
```

### 1. PicList/PicGO（推荐）⭐

#### 为什么推荐

- ✅ 支持 20+ 种图床（阿里云、七牛云、腾讯云、AWS S3、MinIO、GitHub、Imgur 等）
- ✅ 配置一次，多处使用（浏览器插件、命令行工具都可以用）
- ✅ 图床配置在 PicList 中管理，插件无需重复配置
- ✅ 图床迁移只需在 PicList 切换配置
- ✅ 支持图片压缩、重命名、水印等插件扩展

#### 配置步骤

**第一步：安装 PicList**

1. 访问 [PicList 官网](https://piclist.cn/) 下载安装
2. 或使用 PicGO（功能类似）

**第二步：配置 PicList 的图床**

1. 打开 PicList，进入「图床设置」
2. 选择你要使用的图床（如阿里云 OSS）
3. 填入认证信息（AccessKey、SecretKey 等）
4. 点击「确定」保存

**第三步：启用 PicList Server**

1. 打开 PicList，进入「PicGo 设置」
2. 找到「设置 Server」，勾选「开启 Server」
3. 默认端口为 36677，可自定义
4. 复制 Server 地址：`http://127.0.0.1:36677/upload`

**第四步：配置插件**

1. IDE 中打开 `Settings/Preferences` → `Markdown Image Kit`
2. 图床服务商选择「PicList」
3. 填入 API 地址：`http://127.0.0.1:36677/upload`
4. （可选）指定图床类型：如 `aliyun`、`qiniu` 等
5. （可选）指定配置名称：如 `default`、`backup`
6. （可选）填入接口密钥：如果在 PicList 中设置了鉴权
7. 点击 `Test` 按钮测试连接
8. 勾选「设置为默认图床」
9. 点击「Apply」保存

#### 配置项说明

| 配置项        | 说明                | 示例                              | 是否必填 |
|------------|-------------------|---------------------------------|------|
| **API 地址** | PicList Server 地址 | `http://127.0.0.1:36677/upload` | ✅ 必填 |
| **图床类型**   | PicList 配置的图床名称   | `aliyun`、`qiniu`、`aws-s3`       | ❌ 选填 |
| **配置名称**   | 多配置时指定使用哪个        | `default`、`backup`              | ❌ 选填 |
| **接口密钥**   | API 鉴权密钥          | -                               | ❌ 选填 |
| **CLI 路径** | PicList 可执行文件路径   | `/Applications/PicList.app`     | ❌ 选填 |

**配置项详解**：

##### API 地址

- 默认值：`http://127.0.0.1:36677/upload`
- 说明：PicList Server 监听的地址和端口
- 如何获取：在 PicList 的「PicGo 设置」→「设置 Server」中查看

##### 图床类型（可选）

- 说明：当 PicList 配置了多个图床时，指定使用哪个
- 可选值：`aliyun`、`qiniu`、`tencent`、`aws-s3`、`github`、`imgur` 等
- 留空：使用 PicList 的默认图床

##### 配置名称（可选）

- 说明：当同一个图床配置了多个账号时，指定使用哪个配置
- 示例：`default`、`backup`、`personal`、`company`
- 留空：使用该图床的默认配置

##### 接口密钥（可选）

- 说明：如果在 PicList 中启用了 API 鉴权，需要填入密钥
- 如何设置：在 PicList 的「PicGo 设置」→「设置 Server」中配置
- 留空：表示无需鉴权

##### CLI 路径（可选）

- 说明：PicList 可执行文件的路径，用于命令行方式上传
- 场景：当 Server 方式不可用时，可以使用 CLI 方式
- 留空：只使用 Server 方式

**步骤一：打开图床迁移对话框**

- 右键点击 Markdown 文件或目录
- 选择 `Markdown Image Kit` → `图床迁移`

**步骤二：填写迁移配置**

1. **输入源图床域名**：
    - 输入要迁移的图床域名（如 `old-cdn.com`）
    - 输入 `ALL` 表示迁移所有网络图片
2. **选择目标图床**：
    - 从下拉框选择目标图床
    - 或选择「本地存储」下载到本地

**步骤三：执行迁移**

- 点击「OK」开始迁移
- 等待迁移完成

### 迁移场景

#### 场景一：图床到图床迁移

**需求**：原图床到期，迁移到新图床

**操作步骤**：

1. 配置新图床并设为默认
2. 打开图床迁移对话框
3. 输入源图床域名：`old-cdn.com`
4. 选择目标图床：如「PicList」
5. 点击「OK」执行迁移

**迁移效果**：

```markdown
# 迁移前
![](https://old-cdn.com/images/image.png)

# 迁移后
![](https://new-cdn.com/images/image.png)
```

#### 场景二：图床到本地迁移（新功能）⭐

**需求**：图床到期且暂时没有其他图床可用，先将图片保存到本地

**操作步骤**：

1. 打开图床迁移对话框
2. 输入源图床域名：`old-cdn.com` 或 `ALL`
3. 选择目标图床：**「本地存储」**
4. （如果未配置全局存储路径）输入存储路径：如 `./imgs`
5. 点击「OK」执行迁移

**迁移效果**：

```markdown
# 迁移前
![](https://old-cdn.com/images/image.png)

# 迁移后
![](./imgs/image.png)
```

**存储路径说明**：

插件会按以下优先级确定图片存储路径：

##### 1. 优先使用全局配置路径

如果在「图片处理设置」→「插入图片时的操作」中已配置了存储路径，插件会自动使用该路径。

**全局配置的路径包括**：

- 复制图片到当前文件夹（`./`）
- 复制图片到 ./assets 文件夹（`./assets`）
- 复制图片到 ./${filename}.assets 文件夹
- 复制到指定路径（自定义路径）

**优点**：

- ✅ 无需额外输入，直接开始迁移
- ✅ 保持与日常使用的路径一致
- ✅ 适合有固定图片存储规范的场景

##### 2. 临时存储路径（仅本次迁移生效）

如果未配置全局存储路径（如选择了「无特殊操作」或「上传图片」），迁移对话框会显示「存储路径」输入框，要求你输入一个临时路径。

**临时路径的特点**：

- 🔸 **仅本次迁移有效**：路径只在当前这次迁移任务中使用
- 🔸 **不影响全局配置**：输入的路径不会保存到设置页面
- 🔸 **灵活性高**：可以根据不同的迁移任务使用不同的路径
- 🔸 **默认值**：输入框默认显示 `./imgs`，可以直接使用或修改

**临时路径示例**：

```
相对路径：
- ./imgs                    # 保存到当前目录的 imgs 子文件夹
- ./backup                  # 保存到当前目录的 backup 子文件夹
- ../shared/images          # 保存到上级目录的 shared/images 文件夹

绝对路径：
- /Users/username/Pictures  # macOS 绝对路径
- D:/Images                 # Windows 绝对路径
```

**使用场景对比**：

| 场景                | 是否需要输入临时路径 | 说明                 |
|-------------------|------------|--------------------|
| 已设置「复制到 ./assets」 | ❌ 不需要      | 直接使用 `./assets` 路径 |
| 已设置「上传图片」         | ✅ 需要       | 因为没有本地路径配置，需要输入    |
| 已设置「无特殊操作」        | ✅ 需要       | 因为没有本地路径配置，需要输入    |
| 已设置「复制到指定路径」      | ❌ 不需要      | 使用你配置的自定义路径        |

**实际操作示例**：

**示例一：已配置全局路径**

```
设置页面配置：插入图片时的操作 = 复制图片到 ./assets 文件夹

迁移时：
1. 打开图床迁移对话框
2. 输入源图床域名：old-cdn.com
3. 选择目标：本地存储
4. ✅ 不显示存储路径输入框（自动使用 ./assets）
5. 点击 OK 开始迁移
6. 图片保存到：./assets/image.png
```

**示例二：未配置全局路径**

```
设置页面配置：插入图片时的操作 = 上传图片（无本地路径）

迁移时：
1. 打开图床迁移对话框
2. 输入源图床域名：old-cdn.com
3. 选择目标：本地存储
4. ⚠️ 显示存储路径输入框（需要输入临时路径）
5. 输入临时路径：./backup/images
6. 点击 OK 开始迁移
7. 图片保存到：./backup/images/image.png
8. ✅ 设置页面的配置不受影响（仍然是"上传图片"）
```

**临时路径的注意事项**：

- ⚠️ 临时路径仅在本次迁移任务中生效
- ⚠️ 下次迁移时，如果仍未配置全局路径，需要重新输入
- ⚠️ 如果经常需要迁移到本地，建议在设置页面配置全局路径
- ⚠️ 临时路径不会自动创建目录，如果目录不存在会自动创建

##### 3. 路径格式和文件命名

- **支持的路径格式**：相对路径（如 `./imgs`）和绝对路径（如 `/Users/xxx/Pictures`）
- **文件命名规则**：下载的图片会根据「图片增强处理设置」中的「图片重命名」配置生成文件名
- **自动创建目录**：如果指定的目录不存在，插件会自动创建

##### 4. 与全局配置的关系

**重要说明**：临时存储路径与全局配置是完全独立的

```
全局配置路径（设置页面）        临时存储路径（迁移对话框）
        ↓                              ↓
   持久化保存                        本次迁移
   所有操作使用                      仅本次使用
   可以修改                          不保存
```

**例如**：

- 你在设置页面配置了「复制图片到 ./assets」
- 但在某次迁移时输入了临时路径 `./backup`
- 结果：
    - ✅ 本次迁移的图片保存到 `./backup`
    - ✅ 设置页面的配置仍然是 `./assets`（不受影响）
    - ✅ 下次粘贴图片时，仍然保存到 `./assets`（使用全局配置）
    - ✅ 下次迁移时，如果选择本地存储且无全局路径，仍需输入临时路径

**适用场景**：

- ✅ 图床到期，需要紧急备份
- ✅ 外部图片链接不稳定，需要本地化
- ✅ 需要离线使用文档
- ✅ 将博客文章转换为本地文档

**注意事项**：

- ⚠️ 确保网络连接稳定
- ⚠️ 确保有足够的本地存储空间
- ⚠️ 大量图片迁移可能需要较长时间
- ⚠️ 如果图片 URL 无后缀，会从响应头获取 Content-Type 判断格式

#### 场景三：迁移所有网络图片

**需求**：文档中引用了多个不同图床的图片，统一迁移

**操作步骤**：

1. 配置目标图床
2. 打开图床迁移对话框
3. 输入：`ALL`
4. 选择目标图床
5. 点击「OK」执行迁移

**迁移效果**：

```markdown
# 迁移前
![](https://cdn1.com/image1.png)
![](https://cdn2.com/image2.png)
![](https://cdn3.com/image3.png)

# 迁移后（统一迁移到新图床）
![](https://new-cdn.com/image1.png)
![](https://new-cdn.com/image2.png)
![](https://new-cdn.com/image3.png)
```

### 迁移原理

1. 解析 Markdown 文档，识别符合条件的图片标签
2. 下载原图（从源图床）
3. 上传到目标图床（或保存到本地）
4. 替换图片 URL
5. 更新 Markdown 文档

## 💡 使用技巧

### 技巧一：快速上传粘贴的图片

**配置**：

- 「插入图片时的操作」选择「上传图片」
- 「设置为默认图床」勾选

**效果**：

1. 截图或复制图片文件
2. 粘贴到 Markdown 文档（`Ctrl/Cmd + V`）
3. 自动上传到图床 🎉

### 技巧二：本地备份 + 图床发布

**配置**：

- 「插入图片时的操作」选择「复制图片到 ./assets 文件夹」
- 「对本地位置的图片应用上述规则」勾选

**使用**：

1. 粘贴图片时自动保存到 `./assets/`
2. 提交到 Git 仓库作为备份
3. 需要发布时，右键选择「图片上传」批量上传到图床

**优势**：

- ✅ 本地有备份
- ✅ 图床挂了可以重新上传
- ✅ 文档可以离线使用

### 技巧三：统一图片命名规则

**配置**：

- 「图片重命名」勾选
- 「重命名模板」填入：`${datetime:yyyyMMdd}-${string:6}`

**效果**：

- 所有图片统一命名为：`20251102-aB3xK9.png`
- 避免文件名冲突
- 方便按日期归档

### 技巧四：优化图片大小

**配置**：

- 「图片压缩」勾选，质量设置为 75
- 「转为 WebP」勾选，质量设置为 80

**效果**：

- 大幅减少图片文件大小
- 提升页面加载速度
- 节省存储和流量成本

### 技巧五：批量处理文档

**场景**：有很多旧文档需要统一处理

**操作**：

1. 选择目录或多个文件
2. 右键选择 `Markdown Image Kit` → 对应功能
3. 批量执行

**支持的批量操作**：

- 批量上传本地图片
- 批量图床迁移
- 批量标签替换

## ❓ 常见问题

### Q1：图片上传失败？

**可能原因**：

- ❌ 网络连接问题
- ❌ 图床配置错误
- ❌ 图床服务不可用
- ❌ 图片格式不支持

**解决方法**：

1. 点击「Test」按钮测试图床配置
2. 查看 MIK Console 日志（开启「MIK Console 日志输出」）
3. 检查网络连接
4. 查看图床服务状态

### Q2：相对路径图片无法显示？

**可能原因**：

- ❌ Markdown 渲染器不支持相对路径
- ❌ 路径格式不正确

**解决方法**：

1. 开启「为相对路径添加 ./」
2. 开启「插入时自动转义图片 URL」
3. 使用绝对路径（关闭「优先使用相对路径」）

### Q3：批量上传很慢？

**可能原因**：

- 图片数量多
- 图片文件大
- 网络速度慢

**优化方法**：

1. 开启「图片压缩」减小文件大小
2. 分批处理，避免一次处理太多
3. 使用网络条件好的环境

### Q4：图床迁移后原图片还能访问吗？

**回答**：

- ✅ 可以，迁移只是复制，不会删除原图床的图片
- ✅ Markdown 文档已更新为新图床的 URL
- ✅ 如果不再需要原图床，可以手动删除

### Q5：PicList 连接失败？

**检查清单**：

- [ ] PicList 是否正在运行
- [ ] PicList Server 是否已开启
- [ ] 端口号是否正确（默认 36677）
- [ ] 防火墙是否拦截
- [ ] API 地址格式是否正确

### Q6：WebP 格式不显示？

**可能原因**：

- 使用的 Markdown 渲染器不支持 WebP
- 浏览器版本过旧

**解决方法**：

- 关闭「转为 WebP」选项
- 升级浏览器
- 使用支持 WebP 的渲染器
